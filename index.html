IAM — Identity & Access Management

Beginner

What it is

IAM controls WHO can do WHAT on AWS. Think of it as the bouncer system — it decides who gets into the club and which rooms they can access.

  • Users — actual people with long-lived credentials. Avoid for apps.
  • Roles — temporary identity assumed by services, apps, or cross-account. Always prefer roles over users for EC2/Lambda.
  • Groups — collection of users sharing a policy. Cannot be attached to resources.
  • Policies — JSON documents granting or denying permissions.

Exam keywords → IAM Role

  • "EC2 instances need access to S3" → IAM Role + Instance Profile
  • "short-term credentials" → IAM Role (STS)
  • "principle of least privilege" → fine-grained IAM policy
  • "temporary access for contractors" → IAM Role with STS
  • "cross-account access" → IAM Role with trust policy
❌ Common Trap "Attach the IAM policy to the EC2 instances" — WRONG. Policies attach to roles/users/groups, not directly to EC2. You attach a role to EC2 via an Instance Profile.

KMS — Key Management Service

Intermediate

Key Types

  • AWS Managed Keys (aws/rds, aws/s3) — automatic, no control over policy
  • Customer Managed Keys (CMK) — you define who can use them via key policy
  • Customer Provided Keys (SSE-C) — you supply the key on every request, AWS never stores it
  • SSE-S3 — S3 manages the entire encryption lifecycle for you

Exam Pattern: Which encryption to use?

  • "Specific IAM roles must manage keys" → KMS CMK + key policy
  • "Tenant brings their own keys" → SSE-C
  • "Simplest encryption, least overhead" → SSE-S3
  • "Encrypt + rotate automatically" → KMS CMK with auto-rotation
  • "Cross-Region replication with imported keys" → SSE-KMS + CRR

Organizations & SCPs

Advanced

Service Control Policies (SCPs)

SCPs define the maximum permissions for accounts in an OU. They don't grant permissions — they set guardrails.

  • Applied at root/OU/account level
  • Even root user of member account is restricted by SCPs
  • Use "deny all ec2:* at production OU" + "allow RunInstances at dev OU"

Control Tower

Automated landing zone setup for multi-account AWS environments.

  • "Audit all API calls across accounts" → Control Tower + CloudTrail
  • "FSBP compliance across accounts" → Control Tower + Security Hub
  • "New account vending" → Control Tower Account Factory

Lambda — Serverless Compute

Intermediate

Lambda Facts You Must Know

  • Max timeout: 15 minutes
  • Max memory: 10 GB
  • Max deployment package: 250 MB (zip), 10 GB (container)
  • Concurrency: soft limit 1000/region, increase with provisioned concurrency
  • /tmp storage: 512 MB–10 GB

Lambda Exam Triggers

  • "API Gateway timeout" → Decouple with SQS queue
  • "Lambda timeout 15min exceeded" → Use ECS Fargate or Step Functions
  • "Shared libraries across Lambda" → Lambda Layers or EFS mount
  • "Lambda + VPC + RDS" → RDS Proxy (connection management)
  • "Cold start latency" → Provisioned concurrency

S3 Advanced Features

Advanced

S3 Access Points

When S3 bucket policy approaches size limit due to many applications → create an S3 Access Point per application instead of cramming all rules in one bucket policy.

S3 Multi-Region Access Points

"Nearest S3 bucket, no public network congestion, automatic failover" → S3 MRAP in active-active config with CRR (Cross-Region Replication).

S3 Requester Pays

When you want to share large datasets publicly but don't want to pay the egress fees → configure Requester Pays bucket. Requester pays for data transfer.

S3 Presigned URLs

Temporary access to private S3 objects without AWS credentials. Expires after a set time. Used for secure file uploads/downloads from web apps.

DynamoDB Advanced

Advanced

DynamoDB vs ElastiCache

Both cache data but serve different purposes:

  • DAX: in-memory cache specifically for DynamoDB — microsecond latency, key-based lookups
  • ElastiCache Redis: general-purpose cache for RDS, application data, session storage, complex data structures
  • ElastiCache Memcached: simple, horizontal scaling, no persistence

DynamoDB Key Patterns

  • "DynamoDB throttling / ProvisionedThroughputExceeded" → DAX cluster
  • "Order status notifications on change" → DynamoDB Streams + Lambda + SNS
  • "Global multi-region NoSQL" → DynamoDB Global Tables
  • "HTTP 500 errors from DynamoDB" → Exponential backoff + retries

Visibility Timeout Trap (Q342)

Lambda processing SQS messages triggers multiple times → set visibility timeout > function timeout + batch window. This prevents the message from being re-read while Lambda is still processing it.

Also use a Dead-Letter Queue (DLQ) for messages that fail after N retries.

Step Functions

Advanced
When to use Step Functions "Multiple intermediate steps and states" + "store states for future reference" + "specific order of processing" → Step Functions (state machine). Not Lambda alone (Lambda has no built-in state management).

Elimination Strategies

❌ Eliminate These First

  • "Create IAM users for applications" → always wrong (use roles)
  • "Store credentials in code/user data" → always wrong
  • "VPC peering for hundreds of VPCs" → wrong (use Transit GW)
  • "ALB for TCP/UDP millions of requests" → wrong (NLB)
  • "Governance mode to prevent root deletion" → wrong (Compliance mode)
  • "Manual process that adds overhead" → almost always wrong
  • "Modify application code when question says you can't" → wrong

✅ Select These First

  • "Least operational overhead" → managed/serverless service
  • "Most cost-effective" → eliminate over-provisioned solutions
  • "Least development effort" → existing AWS integration, not custom code
  • "High availability" → Multi-AZ + Auto Scaling
  • "Decouple" → SQS (not direct connection)
  • "Existing Kubernetes" → EKS (not re-architect)
  • "Compliance + no deletion" → S3 Object Lock Compliance mode

🔐 Security Questions

Q1. A company's application stores product data and object files in an Amazon S3 bucket. The company recently experienced malicious attacks. The company needs a solution that continuously monitors for malicious activity in the AWS account, workloads, and access patterns to the S3 bucket. The solution must report suspicious activity and display findings on a dashboard. Which solution meets these requirements?
  • A. Configure Amazon Macie to monitor and report findings to AWS Config.
  • B. Configure Amazon Inspector to monitor and report findings to AWS CloudTrail.
  • C. Configure Amazon GuardDuty to monitor and report findings to AWS Security Hub.
  • D. Configure AWS Config to monitor and report findings to Amazon EventBridge.
✅ Answer: C

GuardDuty is the service designed for continuous monitoring of malicious activity across your AWS account, including EC2 network traffic, CloudTrail API calls, and S3 data events. Security Hub provides the centralized dashboard that aggregates GuardDuty findings alongside Inspector, Macie, and other security services.

Why others are wrong:
A: Macie detects PII in S3, not general malicious activity. Config tracks configuration changes, not threats.
B: Inspector scans for software vulnerabilities in EC2/containers. CloudTrail is an audit log, not a dashboard.
D: Config tracks compliance, not malicious activity.
Q2. A company is conducting an internal audit and wants to ensure that its Amazon S3 data lake does not contain PII such as passport numbers and credit card numbers. Which solution meets these requirements?
  • A. Configure AWS Audit Manager with PCI DSS standards.
  • B. Configure S3 Inventory and query with Amazon Athena.
  • C. Configure Amazon Macie to run a data discovery job using managed identifiers for the required data types.
  • D. Use S3 Inventory Report to scan the S3 bucket.
✅ Answer: C

Amazon Macie is specifically built for PII discovery in S3. Its managed identifiers cover 80+ types of sensitive data including passport numbers, credit cards, SSNs, driver's license numbers, etc. — directly matching the requirement.

Why others are wrong:
A: Audit Manager assesses compliance frameworks, not data content scanning.
B: S3 Inventory lists objects and metadata (size, encryption) — not file content.
D: Same issue as B — inventory metadata only, not content scanning.
Q3. A financial services company must keep log data for 1 year in S3. The company wants to prevent any user — including the AWS account root user — from deleting or overwriting the log files for 1 year. The data must be available for read-only requests. Which solution meets these requirements?
  • A. Enable S3 Versioning on the bucket. Use Object Lock in compliance mode with a 1-year retention period.
  • B. Enable S3 Transfer Acceleration. Create a Lifecycle rule to move objects to Glacier after 1 year.
  • C. Enable S3 Versioning. Create a Lifecycle rule to move objects to Glacier after 1 year.
  • D. Create a Lambda function to check timestamps and move data to Glacier Deep Archive if older than 1 year.
✅ Answer: A

The key requirement is "no user including root can delete or overwrite" — this requires S3 Object Lock in Compliance mode. Governance mode allows privileged users to override; Compliance mode is absolute.

S3 Versioning must be enabled to use Object Lock. The 1-year retention period ensures objects cannot be deleted for 12 months. Data remains accessible (read-only) without retrieval delays.

Why others are wrong:
B: Transfer Acceleration is for fast uploads, not immutability. Moving to Glacier means retrieval delays.
C: Lifecycle policy to Glacier means data is not immediately readable — violates read availability requirement.
D: Lambda is custom code overhead; cannot prevent root from deleting anyway.

⚙️ Compute Questions

Q4. A company is planning to migrate a TCP-based application into a VPC. The application must be publicly accessible on a non-standard TCP port and must process up to 3 million requests per second with low latency. Which solution meets these requirements?
  • A. Deploy a Network Load Balancer (NLB). Configure it to be publicly accessible over the required TCP port.
  • B. Deploy an Application Load Balancer (ALB). Configure it to be publicly accessible over the required TCP port.
  • C. Deploy a CloudFront distribution. Use an ALB as the origin.
  • D. Deploy an API Gateway configured with the TCP port. Use Lambda with provisioned concurrency.
✅ Answer: A

NLB operates at Layer 4 (TCP/UDP), handles millions of requests per second, provides ultra-low latency, and supports any TCP port. This is the textbook NLB use case.

Why others are wrong:
B: ALB is Layer 7 (HTTP/HTTPS only). It cannot handle arbitrary non-standard TCP ports at the protocol level.
C: CloudFront also supports only HTTP(S), not arbitrary TCP. Also adds unnecessary latency for a backend API.
D: API Gateway supports HTTP(S) only, not raw TCP. Lambda has a 15-min timeout but this doesn't matter here — the fundamental issue is TCP protocol support.
Q5. A company hosts an application using Amazon EC2 Auto Scaling groups. Peak traffic occurs at the same time every day, and users report slow performance at the START of peak hours. The app performs normally 2-3 hours after peak begins. Which solution meets these requirements?
  • A. Configure an ALB to distribute traffic properly to the instances.
  • B. Configure dynamic scaling based on memory utilization.
  • C. Configure dynamic scaling based on CPU utilization.
  • D. Configure a scheduled scaling policy to launch new instances before peak hours.
✅ Answer: D

The problem is that dynamic (reactive) scaling takes 2-3 hours to respond — by then traffic has already been slowing users down. Since peak hours are predictable and consistent daily, Scheduled Scaling pre-provisions instances BEFORE the peak arrives.

Why others are wrong:
A: An ALB distributes traffic but doesn't add capacity — the problem is insufficient instances, not distribution.
B & C: Dynamic scaling reacts to metrics AFTER they spike. By the time CPU/memory rises enough to trigger scaling and new instances boot, 2-3 hours have passed. This describes exactly the symptom — dynamic scaling is too slow for predictable peaks.

🗄️ Storage Questions

Q6. A solutions architect needs storage for a video editing application running on Amazon Linux EC2 instances. The application requires sub-millisecond latency, access to 500 TB of video data, and data must be available from multiple EC2 instances simultaneously. Which solution meets these requirements?
  • A. Deploy an S3 bucket with S3 access points.
  • B. Configure an FSx for Lustre file system with SSD storage.
  • C. Deploy a Provisioned IOPS SSD (io2) EBS volume.
  • D. Configure an EFS file system in Max I/O performance mode.
✅ Answer: B

FSx for Lustre is designed for high-performance compute workloads requiring sub-millisecond latency at massive scale (petabytes) with concurrent access from multiple Linux instances. It's the go-to for HPC, ML, and media processing.

Why others are wrong:
A: S3 doesn't offer sub-millisecond latency — it's object storage with ms–s latency, not a file system.
C: EBS io2 only attaches to ONE EC2 instance at a time (unless using Multi-Attach with very limited support). Cannot share 500 TB across multiple instances simultaneously.
D: EFS Max I/O mode increases throughput but at the cost of higher latency. It's optimized for thousands of concurrent connections, not sub-millisecond media processing.
Q7. A company stores raw data in S3. Analytics on the data are unpredictable — the company cannot predict or control access patterns. The company wants to reduce S3 costs. Which solution meets these requirements?
  • A. Use S3 replication to transition infrequently accessed objects to S3 Standard-IA.
  • B. Use S3 Lifecycle rules to transition from S3 Standard to Standard-IA.
  • C. Use S3 Lifecycle rules to transition objects from S3 Standard to S3 Intelligent-Tiering.
  • D. Use S3 Inventory to identify objects not accessed and manually move them.
✅ Answer: C

The key phrase is "cannot predict or control the access pattern." S3 Intelligent-Tiering is specifically designed for this scenario — it automatically moves objects between frequent, infrequent, and archive tiers based on actual access patterns, without retrieval fees.

Why others are wrong:
A: Replication doesn't change storage class on the original bucket, and Standard-IA would be wrong for unpredictable access (you'd be charged extra retrieval fees for unexpectedly frequent access).
B: Lifecycle rules to Standard-IA assume you KNOW when objects become infrequent. If access is unpredictable, you'd end up paying Standard-IA retrieval fees on frequently accessed objects.
D: Manual process adds operational overhead, and S3 Inventory doesn't analyze access patterns in real time.

🌐 Networking Questions

Q8. A company needs to connect several VPCs in us-east-1 that span hundreds of AWS accounts. The company's networking team has its own account to manage the cloud network. What is the MOST operationally efficient solution?
  • A. Set up VPC peering connections between each VPC and update all route tables.
  • B. Configure a NAT gateway and internet gateway in each VPC to connect through the internet.
  • C. Create an AWS Transit Gateway in the networking team's account and configure static routes from each VPC.
  • D. Deploy VPN gateways in each VPC and create a transit VPC.
✅ Answer: C

Transit Gateway is the hub-and-spoke solution for connecting hundreds of VPCs across accounts. You attach each VPC once, manage routing centrally, and it scales massively. This is far more operationally efficient than N² peering connections.

Why others are wrong:
A: VPC peering is non-transitive (A-B + B-C ≠ A-C). For hundreds of VPCs, you'd need N*(N-1)/2 peering connections. At 100 VPCs, that's 4,950 connections — operationally impossible.
B: Routing through public internet is a security and latency problem. NAT gateways add cost and NAT gateways can't route between VPCs.
D: Transit VPC with VPN gateways is an older, more complex pattern superseded by Transit Gateway. More overhead, more cost.
Q9. An online video game company needs to handle millions of UDP internet traffic requests each second with ultra-low latency. The game servers run on EC2 instances. Which solution meets these requirements MOST cost-effectively?
  • A. Configure an Application Load Balancer with the required protocol and ports.
  • B. Configure a Gateway Load Balancer for the internet traffic.
  • C. Configure a Network Load Balancer with the required protocol and ports.
  • D. Launch identical game servers in separate AWS Regions and route to both.
✅ Answer: C

NLB is Layer 4 and supports UDP natively. It handles millions of requests per second with the lowest latency of all AWS load balancers. This is exactly its use case.

Why others are wrong:
A: ALB is HTTP/HTTPS only — it does NOT support UDP. Gaming traffic on UDP would fail entirely with ALB.
B: GLB is for inline inspection appliances (firewalls, IDS/IPS) — it's transparent passthrough, not for user-facing game traffic distribution.
D: Duplicate servers in multiple Regions doubles cost without addressing the load distribution problem at a single region.

🔗 Integration Questions

Q10. An order-processing microservice must remain responsive and decoupled from the frontend. It must also be able to reprocess orders that fail on the first attempt. Which solution meets these requirements?
  • A. Deploy an ALB in front of the order-processing microservice and auto-scale based on CPU.
  • B. Deploy an SQS queue to integrate frontend and order-processing. Frontend sends messages to queue. EC2 processes messages from queue.
  • C. Establish direct HTTPS connections from frontend to microservice. Use a dynamically expanding thread pool.
  • D. Use Kinesis Data Streams to ingest order requests. EC2 continuously polls the stream.
✅ Answer: B

SQS is the canonical decoupling service. Messages persist in the queue until successfully processed. Failed messages can be retried automatically via visibility timeout or sent to a Dead-Letter Queue (DLQ) for reprocessing. The frontend is decoupled (doesn't wait for the backend to process).

Why others are wrong:
A: An ALB scales the microservice but doesn't decouple it. The frontend still waits for a response. Failed requests are lost.
C: Direct HTTPS connections = tight coupling. If the microservice is overwhelmed, frontend users experience failures. No automatic retry mechanism.
D: Kinesis is for real-time analytics and streaming data, not for job queuing with retry behavior. It doesn't have DLQ support or per-message visibility timeout semantics.

🏛️ Database Questions

Q11. A company's transaction-processing application has a MySQL RDS database. During high load, many connections open and close frequently, causing latency. The solutions architect cannot modify the application code. Which solution will improve performance during high load?
  • A. Upgrade the DB instance to a larger instance type.
  • B. Configure RDS storage autoscaling to increase provisioned IOPS.
  • C. Use Amazon RDS Proxy to pool and share database connections.
  • D. Convert the DB instance to a Multi-AZ deployment.
✅ Answer: C

The root cause is "poor connection handling" — too many connections opening/closing. RDS Proxy maintains a pool of persistent database connections and multiplexes application connections through the pool. Applications get fast connection handoff without opening new DB connections each time — no code changes needed.

Why others are wrong:
A: A larger instance has more capacity but doesn't solve the connection overhead problem. At high load, connection churn is the bottleneck, not raw compute.
B: IOPS scaling addresses disk I/O bottlenecks, not connection management latency.
D: Multi-AZ improves fault tolerance and failover — it doesn't address connection pooling or performance under high connection churn.
Q12. A company's web-based relational database workload has highly variable traffic — high during business hours, and near-zero during nights and weekends. Outside business hours, there are long periods of no database activity. The company wants to optimize operational costs. Which solution will meet these requirements?
  • A. Deploy RDS for MySQL Single-AZ On-Demand. Relaunch with larger instance during spikes.
  • B. Deploy RDS for MySQL Multi-AZ. Purchase Reserved Instances.
  • C. Deploy DynamoDB with provisioned capacity and auto scaling.
  • D. Deploy an Aurora Serverless v2 DB cluster. Configure capacity to scale based on workload.
✅ Answer: D

Aurora Serverless v2 automatically scales database capacity in fine-grained increments (fractions of ACU) from near-zero to maximum, and you only pay for what you use. For workloads with "long periods of no database activity," this is the most cost-optimized relational DB option.

Why others are wrong:
A: Manual resizing is operational overhead. On-Demand RDS charges by the hour even when idle.
B: Reserved Instances require commitment — you pay whether you use it or not. Wrong for "long idle periods."
C: DynamoDB is NoSQL — the question specifies relational data with table relationships. Migrating to DynamoDB would require a significant schema redesign (contradicts "optimize costs" without restructuring).

💡 Advanced Multi-Service Questions

Q13 (Advanced). A company needs to store customer-uploaded images directly from browsers, process them asynchronously into multiple sizes, and serve them at low latency globally. The company needs to reduce coupling and improve performance. Which combination of actions should the architect take? (Select TWO)
  • A. Configure the application to upload images to S3 Glacier Flexible Retrieval.
  • B. Configure the web server to upload original images to Amazon S3.
  • C. Configure the application to upload images directly from each user's browser to S3 using presigned URLs.
  • D. Configure S3 Event Notifications to invoke a Lambda function when an image is uploaded. Use the function to resize the image.
  • E. Create an EventBridge rule to invoke Lambda on a schedule to resize uploaded images.
✅ Answer: C, D

C: Presigned URLs allow browsers to upload directly to S3 — this bypasses the web server entirely, reducing server load and coupling. The web server generates a temporary signed URL; the browser uploads directly.
D: S3 Event Notifications triggering Lambda is the classic serverless image processing pattern. The event fires when an object is created; Lambda resizes asynchronously, fully decoupling processing from upload.

Why others are wrong:
A: Glacier is for archival — images need to be immediately accessible. Glacier has retrieval delays of minutes to hours.
B: Having the web server upload to S3 means images travel through the server twice (user→server→S3), adding latency and coupling.
E: A scheduled Lambda runs periodically regardless of uploads — this adds unnecessary latency (images wait for the next schedule trigger). Event-driven is always better for upload workflows.
Q14 (Advanced). A company has an application with Aurora MySQL running in a single AZ. They want it highly available with minimum downtime, minimum data loss, and least operational effort. Which solution meets these requirements?
  • A. Place EC2 in different Regions. Use Route 53 health checks. Use Aurora Cross-Region Replication.
  • B. Configure Auto Scaling group for multiple AZs. Configure Aurora Multi-AZ. Add an RDS Proxy.
  • C. Configure Auto Scaling group for one AZ. Generate hourly snapshots. Recover from snapshots on failure.
  • D. Configure Auto Scaling group for multiple Regions. Write to S3. Use S3 Event Notifications to write to DB.
✅ Answer: B

Three layers of HA with least effort: Multi-AZ Auto Scaling for EC2 availability + Aurora Multi-AZ for database HA (automatic failover in ~30 seconds) + RDS Proxy to maintain DB connections during failover (reduces app-side reconnection errors).

Why others are wrong:
A: Cross-Region adds complexity and cost. Multi-Region is appropriate for disaster recovery, not basic HA. "Minimum operational effort" rules this out.
C: Single AZ Auto Scaling doesn't protect against AZ failure. Hourly snapshots mean up to 1 hour of data loss (high RPO). Recovery from snapshots means significant downtime (high RTO).
D: Writing to S3 then into DB via S3 Event Notifications is a massively over-engineered anti-pattern with eventual consistency issues.
